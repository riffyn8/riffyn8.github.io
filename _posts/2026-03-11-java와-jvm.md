## 자바란?
썬 마이크로시스템즈(현 오라클)에서 개발한 객체지향 프로그래밍 언어. "한 번 작성하면 어디서든 실행된다(write Once, Run Anywhere)"라는 철학 아래 설계되었다.

### 자바의 특징
- **객체지향 프로그래밍(OOP)**: 클래스와 객체를 기반으로 설계하며 추상화, 캡슐화, 상속, 다형성을 지원한다.
- **플랫폼 독립성**: JVM을 통해 OS에 상관없이 동일한 코드를 실행할 수 있다.
- **자동 메모리 관리**: 가비지 컬렉터(GC)가 불필요한 객첼르 자동으로 정리하여 메모리 누수를 방지
- **멀티스레딩 지원**: 언어 차원에서 스레드 관리 기능을 제공하여 효율적인 병렬 처리가 가능

---

## JVM
자바 애플리케이션과 OS 사이에서 중개자 역할을 수행하는 가상 머신이다. 바이트코드를 실행하고 메모리를 관리하며 플랫폼 독립성을 보장한다. 자바 애플리케이션이 OS에 직접 명령을 내리지 않고 JVM을 거치기 때문에, 어떤 OS에서도 동일한 동작을 보장받을 수 있다.

### JVM의 구성 요소

![jvm](https://www.artima.com/insidejvm/ed2/images/fig5-1.gif){: .normal}

| JVM 구성 요소                     | 설명                             |
|-------------------------------|--------------------------------|
| 클래스 로더(Class Loader)          | 바이트코드를 읽어 들여 메서드 영역에 로드        |
| 런타임 데이터 영역(Runtime Data Area) | 프로그램 실행 시 필요한 데이터를 저장하는 메모리 공간 |
| 실행 엔진(Execution Engine)       | 바이트코드를 기계어로 해석하여 실행            |
| 가비지 컬렉터(Garbage Collector)    | 더 이상 참조되지 않은 객체를 메모리에서 제거      |

#### 클래스 로더(Class Loader)
말 그대로 클래스 파일을 메모리에 올리는 역할을 하며, `로딩(Loading)` → `링크(Linking)` → `초기화(Initialization)` 과정을 거친다. 클래스 로더는 모든 클래스를 한 번에 올리지 않고, 해당 클래스가 참조되는 시점에 동적으로 메모리에 로드한다.

1. **로딩**: 바이트코드(`*.class`)를 읽어 메타데이터를 생성하고 메서드 영역(Method Area)에 저장한다.
2. **링크**: 로드된 클래스 파일을 사용할 수 있도록 준비하는 단계.
   - **검증(Verification)**: 바이트코드가 JVM 명세에 부합하는지, 보안상 안전한지 확인
   - **준비(Preparation)**: 클래스 변수(static)를 위한 메모리를 할당하고 기본값으로 초기화한다. 
   - **분석(Resolution)**: 심볼릭 레퍼런스(이름 기반 참조)를 메서드 영역의 실제 주소인 다이렉트 레퍼런스로 교체
3. **초기화**: `static` 변수에 실제 값을 할당하고 `static` 블록을 실행한다.
   - 부모 클래스가 있다면 부모를 먼저 초기화한 후 자식 클래스를 초기화한다. 클래스 내부에서는 static 변수 선언문과 static 블록이 작성된 순서대로 실행된다.

#### 클래스 로더의 3가지 원칙
클래스 로더는 계층 구조를 가지며 다음 원칙에 따라 동작한다.

| 클래스 로더                  | 역할                         |
|-------------------------|----------------------------|
| Bootstrap ClassLoader   | 최상위, JDK 핵심 클래스 로드         |
| Extension ClassLoader   | 확장 라이브러리 로드                |
| Application ClassLoader | 개발자가 작성한 클래스 및 외부 라이브러리 로드 |

1. **위임 원칙 (Delegation Principle)**: 클래스 로드 요청 시 하위 로더는 상위 로더에게 요청을 먼저 위임한다. 이는 시스템의 핵심 자원이 중복 로드되거나 임의로 변경되는 것을 방지하여 시스템의 무결성을 보호하기 위함
2. **가시성 원칙 (Visibility Principle)**: 하위 로더는 상위 로더가 로드한 클래스를 참조할 수 있지만, 상위 로더는 하위 로더가 로드한 클래스를 참조할 수 없다. 이는 검증되지 않은 외부 라이브러리 등이 시스템 핵심 영역을 침범하거나 의존하는 것을 방지한다. 
3. **유일성 원칙 (Uniqueness Principle)**: 이미 로드된 클래스는 다시 로드하지 않는다. 상위 로더가 로드했다면 하위 로더는 이를 다시 로드하지 않아 클래스의 유일성을 보장한다.

#### 런타임 데이터 영역(Runtime Data Area)
JVM이 프로그램을 실행할 때 사용하는 메모리 영역이다. 실행 엔진이 자바 애플리케이션을 실행하면서 필요한 데이터(클래스 정보, 객체, 변수 등)를 저장해두는 공간.

- **메서드 영역(Method Area)**: 클래스 로더가 읽어 들인 클래스 수준의 정보(클래스 구조, 메서드 데이터 등)와 `static` 변수가 저장되는 곳이다. 모든 스레드가 공유하는 영역
- **힙(Heap)**: `new` 연산자로 생성된 객체(인스턴스)와 배열이 저장되는 공간. 인스턴스 변수들이 여기에 위치하며, 가비지 컬렉터(GC)의 주요 관리 대상
- **스택(Stack)**: 메서드 호출 시마다 해당 메서드만을 위한 공간(Frame)이 생성된다. 지역 변수와 매개변수 등이 저장되며, 스택 자료구조의 특성에 따라 후입선출 방식으로 동작한다. 예를 들어 `main()` 메서드가 가장 먼저 실행되어 쌓이고, 그 위로 호출된 메서드들이 차례로 쌓였다가 실행이 끝나면 역순으로 제거된다. 
- **PC 레지스터(PC Register)**: 현재 실행 중인 JVM 명령의 주소를 저장. 스레드마다 하나씩 존재하며, 다음에 실행할 명령의 위치를 가리키는 포인터 역할을 한다.
- **네이티브 메서드 스택(Native Method Stack)**: 자바 언어 이외의 언어(C, C++)로 작성된 네이티브 코드를 실행하기 위한 메모리 영역이다.

#### 실행 엔진(Execution Engine)
클래스 로더를 통해 메모리에 배치된 바이트코드를 실제로 실행하는 역할을 한다. 실행 엔진은 크게 인터프리터와 JIT 컴파일러 두 가지 방식을 혼합하여 사용한다.

- **인터프리터(Interpreter)**: 바이트코드를 한 줄씩 읽어서 실행. 별도의 컴파일 과정이 없기 때문에 프로그램의 초기 실행 속도는 빠르지만, 동일한 코드가 반복 호출될 때마다 매번 다시 해석해야 하므로 전체적인 수행 속도는 느리다.
- **JIT 컴파일러(Just-In-Time Compiler)**: 인터프리터의 단점을 보완하기 위해 도입되었다. 반복되는 코드를 기계어로 완전히 변환하여 캐싱해 두었다가, 이후에는 변환된 기계어를 직접 실행함으로써 처리 속더를 향상시킬 수 있다.

> **동작 원리:** <br/>
> JVM은 처음에는 인터프리터 방식을 통해 코드를 한 줄씩 읽고 번역하여 실행한다. 이 과정에서 JVM은 각 메서드가 얼마나 자주 호출되는지 카운티하며 최적화 대상을 찾아내는데, 이를 **프로파일링(Profiling)** 이라고 한다. <br/>
> <br/>
> 호출 빈도가 컴파일 임계치를 넘어서면 해당 코드는 **Hot Spot**으로 분류된다. JIT 컴파일러는 이렇게 분류된 코드들을 기계어로 번역하여 캐싱해 둔다. 이후 동일한 코드가 호출되면 인터프리터가 다시 해석하는 대신 캐싱된 기계어를 즉시 실행한다. 다만, 프로그램 실행 중에 코드를 해석하고 저장하는 과정에서 CPU 자원을 소모하므로 초기 구동 시 JIT로 인한 지연이 일어날 수 있다.

## 자바 프로그램의 실행 흐름
1. **컴파일 단계**: 소스 파일(`*.java`)을 자바 컴파일러(`javac`)를 통해 바이트코드(`*.class`)로 변환
2. **메인 메서드 실행**: JVM의 실행 엔진이 프로그램의 시작점인 `main()` 메서드를 호출
3. **클래스 로딩 체크**: 실행 엔진이 코드를 실행하다가 특정 클래스가 필요해지면, 해당 클래스가 **메서드 영역(Method Area)**에 로드되어 있는지를 확인
4. **클래스 로더 동작**: 만약 메서드 영역에 찾는 클래스가 없다면, 클래스 로더가 동작하여 해당 클래스 파일을 찾아 로딩 → 링크 → 초기화 과정을 거쳐 메서드 영역에 적재
5. **런타임 데이터 영역 사용**: 실행 엔진은 로드된 클래스 정보를 바탕으로 명령을 수행한다. 이때 `new` 연산자를 통해 인스턴스가 생성되면 힙 영역에 데이터를 저장하고, 메서드 실행 시 필요한 정보는 스택 등에 저장한다. 
6. **반복 수행 및 실행 최적화**: 실행 엔진은 바이트코드를 읽으며 위 과정을 반복한다. 이때 인터프리터 방식과 JIT 컴파일러 방식을 적절히 혼용하여 실행 속도를 최적화한다.
7. **가비지 컬렉션(GC)**: 프로그램 실행 중 힙 영역에 생성된 객체들 중, 더 이상 참조되지 않아 사용되지 않은 객체들은 가비지 컬렉터가 감지하여 자동으로 메모리에서 제거한다.

> `main()` 메서드가 실행되는 시점부터 런타임(Runtime)이라고 한다. 즉, JVM이 바이트코드를 읽어 메모리에 적재하고 실질적으로 프로그램을 구동하는 단계 </br>
> 컴파일 단계에서는 주로 오타나 자료형 불일치 같은 문법 오류를 발견할 수 있으며, 런타임 시에는 컴파일 단계에서 예측하지 못한 예외(Exception)나 논리적 오류가 발생할 수 있다.
